# Hypothesis Guard — 假設驗證模組

## 目的

在 AI 生成研究假設時，防止以下錯誤：
- 把已知概念當成新發現（novelty 誤判）
- 生成無法被實驗驗證或否證的假設
- 假設的前提與結論之間缺乏邏輯連結
- 忽略已有的反面證據

---

## 檢查量表

AI agent 生成假設後，必須逐項回答以下問題。任一項為「否」則觸發警告。

### 1. Novelty 驗證（最關鍵）

```
□ 是否已用至少 3 組不同關鍵詞搜索相關文獻？
□ 搜索範圍是否涵蓋了相鄰領域（不只限於目標領域）？
□ 是否檢查了 2020 年之後的文獻（避免忽略近期進展）？
□ 是否明確列出「最接近的已有工作」並說明本假設的差異？
□ 差異是實質性的還是僅限於術語重新包裝？
```

**操作方式：**
- 將假設拆解為核心概念（通常 2-4 個）
- 對每個核心概念進行獨立文獻搜索
- 再搜索概念的組合
- 如果組合搜索返回高度相關的結果 → 假設可能不夠新穎
- 記錄搜索過程：用了什麼關鍵詞、找到幾篇相關文獻、最接近的是哪篇

**已知失敗案例：**
AI Scientist v1 在獨立評估中，將 micro-batching for SGD 等已確立的概念誤判為 novel。
根本原因：文獻搜索僅用簡單關鍵詞匹配，未做深層概念比對。

### 2. 邏輯一致性

```
□ 假設是否可以寫成 "If X, then Y" 的形式？
□ X（前提）是否有證據支持或合理推論？
□ 從 X 到 Y 的推論路徑是否清晰？
□ 是否存在已知的反例或反面證據？
□ 如果存在反面證據，是否已說明為何假設仍然成立？
```

**操作方式：**
- 強制將假設重新表述為條件句
- 列出隱含的前提假設（至少 3 個）
- 對每個前提假設做簡短的證據搜索
- 主動搜索反面證據（搜索 "failure of X", "limitation of X", "X does not work"）

### 3. 可測試性

```
□ 這個假設能否透過實驗、觀察、或數據分析被驗證？
□ 驗證所需的數據/資源是否實際可取得？
□ 什麼樣的結果會否證這個假設？（必須能回答這個問題）
□ 驗證的時間和成本是否在合理範圍內？
```

**紅線規則：**
如果無法回答「什麼結果會否證這個假設」→ 這不是一個科學假設，退回重新制定。

### 4. 範圍適當性

```
□ 假設的範圍是否足夠具體（不是 "AI will improve healthcare"）？
□ 假設是否有明確的邊界條件（在什麼情境下成立）？
□ 是否區分了相關性和因果性的宣稱？
```

---

## 輸出格式

假設通過驗證後，應附帶以下摘要：

```
## 假設驗證報告
- 假設聲明：[假設的一句話描述]
- Novelty 評估：[HIGH/MEDIUM/LOW] — [最接近的已有工作及差異說明]
- 搜索紀錄：[使用的關鍵詞] × [找到的文獻數] × [最相關的 3 篇]
- 邏輯評估：[前提列表] → [推論路徑] → [已知反例及回應]
- 可測試性：[驗證方法] + [否證條件]
- 信心等級：🟢/🟡/🔴
```

## 常見陷阱

1. **術語陷阱**：同一個概念在不同領域可能有不同名稱。搜索時使用概念描述而非僅用術語。
2. **時間陷阱**：AI 的訓練資料可能不包含最新的文獻。必須用 web search 補充。
3. **確認偏誤**：AI 傾向於搜索支持假設的證據。必須主動搜索反面證據。
4. **粒度陷阱**：假設太粗（不可測試）或太細（缺乏意義）都是問題。

## Domain-Specific Sub-modules

根據假設涉及的領域，載入對應的 sub-module 以獲得更精確的檢查：

- `hypothesis/biomedical.md` — 臨床可轉譯性、生物學合理性、安全性紅線、evidence level
- `hypothesis/cs-ml.md` — SOTA 檢查、可重現性設計、benchmark 陷阱、contribution 類型
- `hypothesis/social-science.md` — 構念效度、再現性危機、WEIRD 偏誤、效果量校準
