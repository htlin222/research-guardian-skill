# Logic Fallacy Chain — 邏輯謬誤檢查模組鏈

## 定位

這不是一個獨立的 Gate，而是**貫穿所有 Gate 的橫切模組**。
邏輯謬誤可以出現在研究的任何階段——假設生成、文獻論述、實驗設計、結果解讀、論文寫作。
現有的 5 道 Gate 檢查的是「素材對不對」，本模組檢查的是「推理通不通」。

```
                    Logic Fallacy Chain（橫切）
                    ─────────────────────────
                    │    │    │    │    │
                    ▼    ▼    ▼    ▼    ▼
                   G1   G2   G3   G4   G5
                  假設  文獻  實驗  結果  寫作
```

每道 Gate 運行時，都應同時啟動本模組中對應的檢查項。

---

## 模組鏈結構

本模組分為 5 層，從基礎到高階，依序掃描：

```
Layer 1: 形式邏輯檢查    — 推理結構本身是否有效
Layer 2: 因果推理檢查    — 因果宣稱是否有根據
Layer 3: 統計推理檢查    — 從數據到結論的推理是否正確
Layer 4: 論證結構檢查    — 整體論證的架構是否完整
Layer 5: 研究特有謬誤檢查 — 學術研究中常見的推理陷阱
```

每層可獨立運行，但建議串聯執行。
如果時間或算力有限，至少執行 Layer 2 + Layer 3（因果 + 統計），
這兩層涵蓋了 AI 研究輸出中最常見的邏輯錯誤。

## Layer 1: 形式邏輯檢查

掃描推理鏈中是否存在經典的形式邏輯謬誤。

### 檢查清單

```
□ 肯定後件 (Affirming the Consequent)
  結構：If P then Q. Q is true. Therefore P.
  研究中的表現：
    "如果我們的假設正確，應該會看到 X 效果。
     我們確實看到了 X 效果。因此假設正確。"
  ⚠ X 效果可能有多種原因，看到 X 不等於證明假設。
  修正：列出可能產生 X 的其他原因，逐一排除。

□ 否定前件 (Denying the Antecedent)
  結構：If P then Q. P is false. Therefore Q is false.
  研究中的表現：
    "如果用方法 A 可以解決問題，那問題是可解的。
     方法 A 失敗了，所以問題不可解。"
  ⚠ 方法 A 失敗不代表問題不可解，可能方法 B 可以。
  修正：明確限制結論範圍為「方法 A 無效」而非「問題不可解」。

□ 不當全稱推論 (Hasty Generalization)
  結構：X 在少數案例中成立 → X 總是成立
  研究中的表現：
    在 3 個資料集上有效 → "本方法具有普遍適用性"
  ⚠ 少量實驗不能支撐全稱結論。
  修正：限定為 "在測試的 N 個資料集上有效"。

□ 虛假二分法 (False Dilemma)
  結構：只有 A 或 B → 不是 A → 所以 B
  研究中的表現：
    "模型要麼過擬合，要麼欠擬合。
     我們的模型沒有過擬合，所以泛化能力很好。"
  ⚠ 可能存在第三種狀態（如在某些子群體過擬合但整體不明顯）。
  修正：討論中間狀態和邊界情況。

□ 循環論證 (Circular Reasoning)
  結構：P is true because Q. Q is true because P.
  研究中的表現：
    "我們的指標是好的因為模型表現好。
     模型表現好因為指標顯示如此。"
  ⚠ 指標本身需要獨立驗證。
  修正：用外部標準驗證指標的有效性。
```

### 與各 Gate 的對應

| Gate | 最常見的形式邏輯謬誤 |
|------|-------------------|
| Hypothesis Guard | 肯定後件、循環論證 |
| Literature Guard | 不當全稱推論 |
| Experiment Guard | 虛假二分法 |
| Result Guard | 肯定後件、否定前件 |
| Writing Guard | 全部都可能出現 |

## Layer 2: 因果推理檢查

AI 最容易犯的邏輯錯誤類型。從「相關」跳到「因果」是研究中最常見的推理飛躍。

### 檢查清單

```
□ 相關當因果 (Correlation ≠ Causation)
  偵測信號：在觀察性研究中使用以下詞彙 →
    "causes", "leads to", "results in", "produces",
    "due to", "because of", "effect of"
  規則：
    - 觀察性研究 → 只能用 "associated with", "correlated with"
    - RCT → 可以謹慎使用 "effect of"，但需限定條件
    - Meta-analysis of RCTs → 可以使用因果語言，但仍需 hedge
  修正：替換為相關性語言 + 列出可能的混淆變項。

□ 反向因果 (Reverse Causation)
  偵測信號：A → B 的宣稱，但 B → A 也同樣合理
  研究中的表現：
    "社交媒體使用導致焦慮"
  ⚠ 可能是焦慮的人更常使用社交媒體。
  修正：明確討論反向因果的可能性。

□ 第三變項問題 (Confounding)
  偵測信號：宣稱 A → B，但未討論可能的 C
  規則：任何因果宣稱都必須伴隨：
    1. 已控制的混淆變項列表
    2. 未能控制的混淆變項列表（誠實承認）
    3. 殘餘混淆的影響方向估計

□ 劑量-反應忽視 (Dose-Response Neglect)
  偵測信號：二元化表述效果，未討論程度
  研究中的表現：
    "X 技術能改善 Y" → 但不說改善多少、在什麼條件下、有無天花板效應
  修正：報告效果量 + 劑量-反應關係（如有）。

□ 時序混淆 (Temporal Confusion)
  偵測信號：因果宣稱中未明確建立時序
  規則：因 (cause) 必須在果 (effect) 之前
    如果研究設計無法確認時序 → 不能做因果推論

□ 辛普森悖論風險 (Simpson's Paradox Risk)
  偵測信號：整體趨勢與子群體趨勢可能不一致
  規則：報告整體結果時，必須檢查：
    1. 是否有重要的子群體
    2. 子群體的結果方向是否與整體一致
    3. 如果不一致，需要解釋為什麼整體分析仍然有效
```

### 因果推理強度階梯

```
最弱 ─────────────────────────────────── 最強

 單一觀察   相關分析   縱貫研究   準實驗   RCT   RCT Meta-analysis
    │          │         │        │       │          │
    ▼          ▼         ▼        ▼       ▼          ▼
  "可能"   "相關"   "或許導致"  "傾向   "因果    "確立的
  "也許"   "關聯"   "預測"     支持"   證據"    因果關係"

AI 的結論語氣不得超過其研究設計所在的階梯位置。
```

## Layer 3: 統計推理檢查

數字正確但從數字得出的結論錯誤——這是 Result Guard 檢查不到的。

### 檢查清單

```
□ 基率忽視 (Base Rate Neglect)
  偵測信號：報告準確率但未考慮基率
  研究中的表現：
    "模型達到 95% 準確率" → 但如果正樣本只佔 5%，
    全猜負也有 95% 準確率。
  修正：報告精確率、召回率、F1、AUROC 等。

□ 多重比較問題 (Multiple Comparisons)
  偵測信號：做了多次檢定但未校正
  規則：如果 N 個比較，任一個 p < 0.05 的機率 = 1 - 0.95^N
    10 個比較 → 40% 機率出現至少一個假陽性
  修正：報告校正後的 p 值（Bonferroni, FDR 等）。

□ 大數謬誤 (Law of Large Numbers Fallacy)
  偵測信號：超大樣本量下報告統計顯著性
  研究中的表現：
    N = 1,000,000 時，微小差異也會 p < 0.001
  ⚠ 統計顯著 ≠ 實際有意義。
  修正：強制報告效果量 + 實際意義的討論。

□ 倖存者偏誤 (Survivorship Bias)
  偵測信號：分析對象是「成功案例」或「現有樣本」
  研究中的表現：
    "分析 100 家成功的 AI 新創公司" → 忽略了失敗的公司
  修正：討論被排除的樣本及其可能影響。

□ 後見之明偏誤 (Hindsight Bias / HARKing)
  偵測信號：結果看起來「太剛好」支持假設
  HARKing = Hypothesizing After Results are Known
  規則：檢查假設是否在看到數據之前就已經確定
    - 如果有預註冊 → 安全
    - 如果沒有 → 標記為探索性分析

□ 生態謬誤 (Ecological Fallacy)
  偵測信號：從群體層級數據推論個體行為
  研究中的表現：
    "A 國的 AI 投資高且 GDP 高 → AI 投資使企業更賺錢"
  ⚠ 國家層級的相關不能推論到個別企業。
  修正：明確標示分析層級 + 限制結論範圍。

□ 倒果為因的 P 值 (P-value Misinterpretation)
  偵測信號：對 p 值的錯誤解讀
  常見錯誤表述：
    ❌ "有 95% 的機率假設是對的"
    ❌ "有 5% 的機率結果是巧合"
    ❌ "p = 0.06 接近顯著，所以有趨勢"
  正確解讀：
    ✅ "在虛無假設為真的條件下，觀察到此數據或更極端數據的機率為 p"
```

## Layer 4: 論證結構檢查

檢查整體論證的架構是否完整、是否存在結構性問題。

### 檢查清單

```
□ 稻草人論證 (Straw Man)
  偵測信號：「反駁」的對象是否與實際主張一致？
  研究中的表現：
    "有人說 X，但我們發現 Y" → 確認真的有人說 X（且說的就是這個意思）
  修正：精確引用被反駁的主張。

□ 訴諸權威 (Appeal to Authority)
  偵測信號："X 大師說過所以是對的"
  規則：
    - 引用該領域的專家意見 → 可以，但不構成證據
    - 引用非該領域的名人意見 → 不可以
    - 引用專家意見但未說明其立場代表少數觀點 → 不完整

□ 滑坡謬誤 (Slippery Slope)
  偵測信號：從一個溫和的前提跳到極端的結論，中間沒有論證
  研究中的表現：
    "如果不採用我們的方法 → 現有方法會越來越差 → 最終系統會崩潰"
  修正：每一步的跳躍都需要獨立的證據。

□ 合成謬誤 (Composition Fallacy)
  偵測信號：零件好 → 整體好
  研究中的表現：
    "每個模組都通過測試 → 整合後的系統一定好"
  ⚠ 模組間的交互作用可能導致整體表現下降。
  修正：明確區分模組測試和整合測試。

□ 分割謬誤 (Division Fallacy)
  偵測信號：整體好 → 零件好
  研究中的表現：
    "系統整體 accuracy 90% → 每個子任務都表現良好"
  ⚠ 整體 90% 可能是某些子任務 99% 而另一些 50%。
  修正：報告各子任務/子群體的分別表現。

□ Cherry-Picking
  偵測信號：只展示支持結論的證據
  規則：
    - 是否報告了所有實驗結果（包括負面結果）？
    - 是否討論了與結論矛盾的已有文獻？
    - 資料集/指標的選擇是否有偏？

□ 移動門柱 (Moving the Goalposts)
  偵測信號：實驗未達到原始目標，但重新定義成功標準
  研究中的表現：
    原始目標："超越 SOTA"
    結果：未超越
    修改後宣稱："在效率方面有優勢"（但原始目標沒提效率）
  修正：如果修改目標，必須明確承認原始目標未達成。
```

## Layer 5: 研究特有謬誤檢查

AI 做研究時特有的、不屬於經典邏輯謬誤但同樣致命的推理問題。

### 檢查清單

```
□ Benchmark 泛化謬誤
  "在 ImageNet 上表現好 → 在真實世界場景中也會好"
  ⚠ Benchmark 是人工策展的，不代表真實分佈。
  修正：明確列出 benchmark 的局限性 + 與真實場景的差距。

□ 消融 ≠ 必要性
  "消融實驗中移除 X 後效果下降 → X 是必要的"
  ⚠ 可能移除 X 後用 Y 替代也同樣有效。
  修正：區分 "X 在本配置中有貢獻" 和 "X 是唯一的解法"。

□ SOTA 幻覺
  "我們的結果超越了所有 baseline → 我們是 SOTA"
  ⚠ 可能遺漏了更新的或更強的 baseline。
  修正：說明 baseline 的選擇標準 + 最後搜索日期。

□ 規模外推
  "在小資料集上的趨勢 → 放大到大資料集也成立"
  ⚠ Scaling laws 不是線性的，可能有 phase transition。
  修正：明確標示實驗規模 + 討論規模外推的風險。

□ 人類等價謬誤
  "AI 達到人類水準的準確率 → AI 和人類一樣好"
  ⚠ 準確率相同不代表錯誤模式相同。
  AI 可能在人類容易的案例上出錯，在困難的案例上反而正確。
  修正：分析錯誤分佈，不只看整體準確率。

□ 負面結果壓制
  AI 傾向於報告正面結果，壓制失敗的實驗。
  檢查：是否有完整的實驗日誌？是否所有實驗都被報告？
  修正：強制報告所有實驗結果，包括失敗的。

□ 複雜度幻覺
  "方法越複雜 → 效果越好"
  ⚠ 複雜的方法如果只比 simple baseline 好 0.1% → 可能不值得。
  修正：報告複雜度-效能的 trade-off + 與 simple baseline 的比較。
```

## 操作方式：如何在每個 Gate 中啟動 Logic Chain

### 自動觸發規則

```
Hypothesis Guard 啟動時 → 同時運行 Layer 1 + Layer 2
Literature Guard 啟動時 → 同時運行 Layer 4（論證結構）
Experiment Guard 啟動時 → 同時運行 Layer 2 + Layer 3
Result Guard 啟動時   → 同時運行 Layer 2 + Layer 3 + Layer 5
Writing Guard 啟動時  → 運行全部 5 層
```

### Subagent 整合

Logic Chain 的評估同樣遵循 subagent 隔離原則：
- 由獨立的 Logic Evaluator Subagent 執行
- 不接收主 agent 的推理過程
- 只接收待評估的輸出 + 本文件的 checklist
- 輸出標準化的「邏輯謬誤報告」

### Logic Evaluator Subagent 的 System Prompt 範本

```
你是一個專門偵測邏輯謬誤的獨立評估者。
你的任務是檢查以下研究輸出中的推理是否有效。

你必須檢查以下層級的謬誤：
[插入本次啟動的 Layer checklist]

規則：
- 對每個可能的謬誤，引用原文中的具體句子作為證據。
- 區分「確定的謬誤」和「可能的謬誤」。
- 對每個確定的謬誤，提供具體的修正建議。
- 不要過度挑剔——普通的修辭簡化不算謬誤，
  只有實質影響結論有效性的推理錯誤才算。
- 沒有發現謬誤也是有效的結論——不需要強迫找問題。
```

```
- [同上格式]

- [同上格式]

- [同上格式]

```

## 嚴重度分級

| 等級 | 定義 | 處理方式 |
|------|------|---------|
| CRITICAL | 推理斷裂導致核心結論無效 | 退回，不可發佈 |
| MAJOR | 重要支持論點有邏輯問題 | 必須修正後才能通過 |
| MINOR | 措辭不精確但不影響核心結論 | 建議修正，可通過 |
| STYLE | 表述習慣而非邏輯錯誤 | 記錄但不阻擋 |

只有 CRITICAL 和 MAJOR 會觸發 Gate 的 FAIL 判定。

### 嚴重度校準規則
```
以下項目在驗證測試中被發現嚴重度偏低，已上調：

Single dataset validation = MAJOR（ML 研究中這是重大限制，不是 MINOR）

未報告的資訊 = 默認 SUSPICIOUS（需要 checklist 主動標記，不是默認 PASS）
```
MINOR 和 STYLE 觸發 WARN，附帶修正建議但不阻擋 pipeline。
